# Threxa — Product Requirements Document

The agent that stops data breaking changes before they merge.

## Overview

Every data team knows the incident: someone renames a column or edits a dbt model, the PR looks harmless, it merges, and three days later the revenue dashboard is wrong and nobody knows why. Impact analysis exists inside catalogs like DataHub, but nobody opens the catalog while reviewing a PR.

Threxa closes that gap. It is an AI agent that sits on the pull request itself. When a PR touches a data model, Threxa parses the change, queries DataHub lineage through the MCP server to compute the full downstream blast radius (tables, dashboards, ML features, and their owners), and posts a verdict on the PR: what breaks, who gets hurt, how to fix it. When a risky change merges anyway, Threxa writes the change record back into DataHub so the catalog remembers what happened and why.

Built for the DataHub Agent Hackathon (deadline Aug 10, 2026). Primary track: Metadata-Aware Code Generation & Development. Secondary framing: Agents That Do Real Work.

## User Roles

- **PR author** (data/analytics engineer): gets an instant blast-radius review instead of finding out in production.
- **Downstream owner** (BI/ML/data product owner): gets named and warned in the PR before their asset breaks.
- **Data platform lead**: sees the history of analyzed changes and near-misses in the dashboard.
- **Hackathon judge**: clicks through a live demo repo + dashboard, watches the agent catch a breaking PR end to end.

## Core Features (MVP)

### F1 — GitHub PR gate (webhook receiver)
GitHub webhook (PR opened/synchronized) hits Threxa's API. Threxa filters for data-relevant files (.sql, dbt models, schema files).
Acceptance: opening a PR with a SQL change triggers an analysis run within seconds; non-data PRs are ignored.

### F2 — Change parser (agent step 1)
Claude parses the diff and extracts structured change intents: entities touched, columns added/renamed/dropped, type changes, logic changes.
Acceptance: dropping a column in a dbt model yields a structured `{entity, column, change_type}` record; parse works on real dbt-style SQL.

### F3 — Blast-radius engine (agent step 2)
For each touched entity, Threxa queries DataHub via the official MCP server: downstream lineage (multi-hop), entity types, owners, usage/quality signals. It builds the impact graph and scores severity: SAFE / RISKY / BREAKING.
Acceptance: for a showcase-ecommerce entity with known downstream consumers, Threxa lists every downstream asset with owner and hop distance; severity reflects whether dropped/renamed columns are actually consumed downstream.

### F4 — PR verdict comment
Threxa posts one rich PR comment: severity badge, impact table (asset, type, owner, hop), a plain-language explanation, and a suggested fix or migration path (generated by Claude with the DataHub schema context, so the fix references real column names).
Acceptance: comment appears on the PR within one run; regenerated (not duplicated) on new pushes.

### F5 — Catalog write-back (context graph contribution)
On analysis completion, Threxa writes back to DataHub: a structured change-record as institutional memory on the touched entities, and on merge of a BREAKING PR, an incident on the affected downstream assets. This is the judging criterion "contribution to the context graph" made literal.
Acceptance: after a run, the DataHub entity page shows Threxa's change documentation; merged breaking PRs create visible incidents.

### F6 — Dashboard
Next.js dashboard: run history, per-run blast-radius view (impact graph), severity stats, and a live "guarded repos" list. Public read access for judges.
Acceptance: every webhook run appears with its verdict and impact detail; no auth wall.

### F7 — Demo fixture
A public demo repo (`threxa-demo-pipeline`) with dbt-style models matching the showcase-ecommerce catalog entities, plus scripted breaking/safe PRs for the video and judge reproduction.
Acceptance: one command / documented click path reproduces the full flow.

## KPIs

- Agent catches 100% of scripted breaking changes in the demo fixture with zero false SAFE verdicts.
- End-to-end latency (PR open → comment) under 60 seconds.
- Every analysis run contributes at least one write-back aspect to DataHub.
- Judge reproduction path under 10 minutes from README.

## Out of Scope (MVP)

- Multi-warehouse ingestion (we use DataHub's showcase-ecommerce datapack as the catalog fixture; the connector path is documented, not rebuilt).
- Production auth/multi-tenancy. Dashboard is public read.
- Full dbt manifest parsing. We parse SQL diffs directly with the agent, which is the demoable core.
- Column-level lineage computation. We consume the lineage DataHub already has.

## Submission Requirements Mapping

- Public repo, Apache 2.0: done at scaffold time.
- Uses DataHub + required component: official DataHub MCP Server (agent tool calls), plus write-back via SDK/API (Agent Context Kit).
- Video under 3 min: scripted around the breaking-PR story.
- Judges test free: hosted dashboard + public demo repo + quickstart-reproducible catalog.
- Bonus: one upstream OSS contribution to datahub-skills or mcp-server-datahub during the build.
